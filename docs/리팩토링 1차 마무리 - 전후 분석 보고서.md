# 리팩토링 1차 마무리 — 전후 분석 보고서

> 대상: `approval`(결재) 도메인 중심, 공통(`common`)·`user`·일부 `notice`/`schedule`
> 범위: 태그 `refactor/start` → 1차 마무리 시점
> 원칙: **겉보기 동작 100% 보존(리팩토링)** — 특성화 테스트가 전 구간 green을 유지하며 이를 증명

---

## 1. 요약 (한 장)

레거시 `ApprovalService`는 검증·영속화·메일·감사로그·권한판정·상태전이·포맷팅을 혼자 다 하는 **God Class**였다.
1차 리팩토링에서 이를 **계층 분리 → 매직넘버 enum화 → DTO 경계 → God Class 해체(도메인 이동·DI·Rich Domain) → 공통 중복 제거 → 네이밍 정리** 순으로 걷어냈다.

- **관찰 동작**: 변경 0. 특성화 테스트 6개가 정수 리터럴(0/1/2/3/9) 기준으로 처음부터 끝까지 green.
- **안전망 강화**: 특성화 6개 + 신규 단위 테스트 10개 = **총 16개** green.
- **상태 변경의 단일 통로**: `status = …` 대입이 이제 `Approval` 도메인 4개 메서드 안에서만 발생(서비스 0건).

## 2. 기법별 전후 비교

| # | 스멜 | Before | After | 기법 |
|---|---|---|---|---|
| 1 | God Class | `ApprovalService`가 7가지 책임 | 오케스트레이션만(저장·메일·감사로그 조율) | 책임 분리 |
| 2 | Long Method | `processApproval` 100줄·중첩 if 6단 | guard clause + `switch` + 4개 private 위임 | Extract Method |
| 3 | Magic Number | status 0/1/2/3/9·type·priority·role·action 정수 | enum + `@Convert`(DB엔 정수 그대로) | Replace Magic Number w/ Enum |
| 4 | Primitive Obsession | 모든 분기를 int 비교 | 내부는 enum, 경계 getter/setter만 int(계약 보존) | Encapsulate |
| 5 | Tight Coupling | `new SmtpMailSender()`·`new FileAuditLogger()` | `MailSender`/`AuditLogger` 인터페이스 생성자 주입 | DIP |
| 6 | Feature Envy | 서비스가 `statusLabel`·`amountGrade` 계산 | `Approval.statusLabel()`·`amountGrade()`(+`AmountGrade` enum) | Move Method |
| 7 | Duplicated Code | 타임스탬프+감사로그 조립 9곳 복붙 | `AuditLogger.write(action,id,userId)` 1곳 소유 | Extract / Pull Up |
| 8 | Anemic Domain | 상태전이·권한이 서비스 if-지옥 | `Approval.submit/approve/reject/cancel`(위반 시 `false`) | Rich Domain / Tell-Don't-Ask |
| 9 | Poor Naming | `d/u/s/proc/tmp` 약어 · 제네릭 enum명 | 의미 식별자 · `ApprovalPriority`/`UserRole` 도메인명 | Rename |
| 10 | Entity 직접 노출 | Map/Entity를 API에 노출 | `record` DTO(요청/응답), JSON·정수 status 보존 | DTO 경계 |

### 대표 코드 전후 (승인 경로)

**Before** — 서비스가 상태를 묻고(ask) 직접 바꿈, 메일·로그 인라인:
```java
if (approval.getStatus() == 1) {
    if (actor.getRole() >= 2) {
        approval.setStatus(2);
        String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
        audit.write("[" + now + "] APPROVAL APPROVE id=" + ... + " by=" + ...);
        ... 메일 본문 인라인 조립 + send ...
}}
```
**After** — 서비스는 시키고(tell) 성공 시에만 부수효과, 조립은 협력 객체가 소유:
```java
private void approve(Approval approval, User actor, Long userId) {
    if (!approval.approve(actor)) return;              // 전이·권한은 도메인이 결정
    repo.save(approval);
    ... mail.send(drafter.getEmail(), "[결재승인] "+approval.getTitle(), approvedBody(...));
    audit.write("APPROVAL APPROVE", approval.getId(), userId);   // 포맷은 AuditLogger가 소유
}
```

## 4. 불변(계약·동작) 보존 증거

리팩토링의 정의 = 내부만 개선, 외부 관찰값 불변. 아래를 **끝까지 지켰다**.

- **API 엔드포인트**: `/api/approvals/{id}/process` 등 URL·HTTP 메서드 그대로.
- **public 시그니처**: `processApproval(id, userId, action, reason)`, `getStatus()/getPriority()/getRole()` int 그대로.
- **DB 저장값**: enum 도입 후에도 `@Convert`로 정수(0/1/2/3/9, 1~4, 1~3) 그대로 저장. `@Enumerated` 미사용.
- **JSON**: `record` DTO가 레거시 필드명·정수 status 그대로 직렬화.
- **동작**: 가드 위반 시 예외가 아니라 `return false` → "조용히 무시"하던 레거시 동작 보존.
- **검증**: 특성화 테스트 6개를 **한 줄도 고치지 않고**(assert·기대값·정수 리터럴 불변) 전 구간 green.
  단, `@Import` 클래스 참조 3줄만 rename을 따라 갱신(테스트 로직 아님, 배선).

## 6. 남은 과제 (2차 — 리팩토링 완료 후 별도 단계)

- **`notice`/`schedule` 도메인화**: 아직 레거시(상태전이 서비스 if, `setStatus(0/1)` 인라인). Approval처럼 Rich Domain 전환.
- **권한 판정 복붙 잔재**: `NoticeService`의 `role >= UserRole.MANAGER.code()`를 `user.isManagerOrAbove()` 호출로 통일(S4).
- **`create()` 초기 상태 세팅**을 도메인 팩토리로 이동 검토.
- **재설계 단계(동작 변경 허용)**: "권한 없는 승인을 조용히 무시" 같은 결함을 예외로 전환 — 이는 리팩토링이 아니라 별도 단계.

> 참고: `setStatus(0)`(초기 상태), 도메인별 `statusLabel` 등은 **우연 중복**이라 통합하지 않는다(도메인마다 상태체계가 독립적으로 진화).

## 7. 결론

1차 리팩토링으로 `approval` 도메인의 주요 스멜(God Class·Long Method·Magic Number·Tight Coupling·Feature Envy·Duplicated Code·Anemic Domain·Poor Naming)을 제거했고,
**관찰 동작은 특성화 테스트로 불변임을 증명**했다. 코드는 이제 "서비스=조율, 도메인=규칙, 협력객체=교체 가능"으로 책임이 정렬되어, 2차(재설계·타 도메인 확산)의 기반이 마련됐다.
