# 4-9. 강결합 탐지

서비스 계층이 생성자 주입(DI) 없이 **협력 객체를 직접 `new` 하는 지점**을 모두 찾는다.
[4-5. 중복 코드 탐지](./4-5.%20중복%20코드%20탐지.md) 2절에서 이미 언급한 내용을 실제 클래스·필드
단위로 구체화하고, DIP(의존성 역전) 리팩토링의 대상 목록으로 정리한다.

## 범위 — 무엇을 "강결합"으로 셀 것인가

같은 `new` 라도 성격이 다르다.

- **협력 객체(다른 클래스의 서비스) 직접 생성** → 이 문서의 대상. 테스트에서 대체 불가능하고,
  Spring이 생명주기를 관리하지 못한다.
- **도메인 엔티티 생성**(`new Approval()`, `new Notice()`, `new Schedule()`, `new User(...)`) → 대상
  아님. `create()` 류 메서드가 자신의 책임으로 새 레코드를 만드는 정상적인 객체 생성이며, DI로 대체할
  성격이 아니다.
- **`DataSourceBuilder.create()`**(`DataSourceConfig.java`) → 대상 아님. `@Configuration` 클래스가
  `@Bean` 메서드 안에서 빈을 조립하는 것은 Spring 표준 패턴 그 자체다.

## 1. 강결합 지점 목록

| 파일 | 줄 | 필드명 | 생성 대상 클래스 | 인터페이스 존재 |
|---|---:|---|---|:---:|
| `ApprovalService.java` | 37 | `mail` | `SmtpMailSender` | ❌ 없음 |
| `ApprovalService.java` | 38 | `audit` | `FileAuditLogger` | ❌ 없음 |
| `NoticeService.java` | 24 | `mail` | `SmtpMailSender` | ❌ 없음 |
| `NoticeService.java` | 25 | `audit` | `FileAuditLogger` | ❌ 없음 |
| `ScheduleService.java` | 20 | `audit` | `FileAuditLogger` | ❌ 없음 |

총 5개 지점, 대상 클래스는 `SmtpMailSender`(2곳)·`FileAuditLogger`(3곳) 2종류.
`ScheduleService` 는 메일을 쓰지 않으므로 `audit` 만 강결합되어 있다.

```java
// ApprovalService.java 36-38행 / NoticeService.java 23-25행과 동일 패턴
private final SmtpMailSender mail = new SmtpMailSender();
private final FileAuditLogger audit = new FileAuditLogger();
```

## 2. 같은 클래스 안에서의 불일치 — repo는 주입, mail/audit는 직접 생성

세 서비스 모두 `Repository` 계열 의존성은 생성자 주입을 정상적으로 받는다.

```java
// ApprovalService.java 40-43행
public ApprovalService(ApprovalRepository repo, UserRepository userRepo) {
    this.repo = repo;
    this.userRepo = userRepo;
}
```

즉 **같은 클래스, 같은 서비스 계층 안에서도 의존성 주입 방식이 두 갈래로 나뉘어 있다** —
`repo`/`userRepo`는 생성자 인자로 받고, `mail`/`audit`는 필드에서 직접 `new`. 이 비일관성 자체가
"어디까지가 관리되는 의존성인가"를 코드만 봐서는 예측하기 어렵게 만든다.

## 3. `SmtpMailSender`/`FileAuditLogger` 자체의 문제

두 클래스 모두 `@Component`/`@Service` 등 Spring 빈 등록 어노테이션이 없는 순수 POJO다
(`common/SmtpMailSender.java`, `common/FileAuditLogger.java`). 즉 문제가 이중으로 겹쳐 있다.

1. **호출부의 문제**: 서비스가 생성자 주입 대신 직접 `new` 한다.
2. **클래스 자체의 문제**: 애초에 Spring 컨테이너에 등록되어 있지 않아, 주입받고 싶어도 지금 상태로는
   `@Autowired`/생성자 주입이 불가능하다(인터페이스도 없어 구현체 교체도 불가능).

두 문제를 함께 해결해야 실제로 테스트 대체(mock/fake)가 가능해진다 — ①인터페이스 추출, ②구현체에
`@Component` 부여, ③호출부를 생성자 주입으로 변경.

## 4. 영향 범위

- **테스트 불가능**: `mail.send()`가 실제로 콘솔에 출력까지 실행되므로, 이 세 서비스는 단위 테스트에서
  메일 발송을 검증하거나 무력화할 방법이 없다(현재 테스트 코드 자체가 0건이라 아직 드러나지 않았을 뿐).
- **중복과 직결**: [4-5. 중복 코드 탐지](./4-5.%20중복%20코드%20탐지.md) 1절의 "감사 로그 기록 패턴
  6곳 중복"은, `audit` 인스턴스가 서비스마다 따로 생성되어 공통 로직을 모을 지점이 애초에 없었기
  때문에 생긴 결과이기도 하다 — 강결합이 중복의 원인 중 하나.
- **런타임 동작 변경 없음**: `SmtpMailSender`/`FileAuditLogger` 는 상태가 없는 클래스라 인스턴스를
  여러 개 만들어도 현재는 관찰 가능한 버그로 이어지지 않는다. 리팩토링 우선순위 판단 시 "동작 결함"이
  아니라 "설계/테스트 용이성 결함"으로 분류해야 한다.

## 5. 요약

| 대상 클래스 | 강결합 지점 수 | 호출 서비스 |
|---|---:|---|
| `SmtpMailSender` | 2 | Approval, Notice |
| `FileAuditLogger` | 3 | Approval, Notice, Schedule |

CLAUDE.md의 "직접 `new` → 생성자 주입(DIP)" 개선 방향에 정확히 대응하는 5개 지점이다. `Repository`
계열은 이미 정상적으로 생성자 주입되어 있으므로, 리팩토링 시 동일한 패턴을 `mail`/`audit` 필드에도
적용하면 된다 — public 메서드 시그니처나 관찰 가능한 동작은 바뀌지 않는 순수 내부 구조 개선이다.
