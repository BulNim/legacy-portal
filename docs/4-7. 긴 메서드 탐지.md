# 4-7. 긴 메서드 탐지

코드베이스 전체 메서드를 대상으로 **줄 수(50줄 초과 여부)** 와 **중첩 깊이** 를 측정해 표로 정리한다.
[4-4](./4-4.%20코드베이스%20구조%20파악.md), [4-5](./4-5.%20중복%20코드%20탐지.md) 에 이어
Long Method / 깊은 중첩(Long Method 계열 스멜)의 실제 지점을 구체화하는 문서.

## 측정 기준

- **줄 수**: 메서드 시그니처 줄부터 닫는 `}` 줄까지 포함한 줄 수(주석·빈 줄 포함, 실제 파일 기준).
- **중첩 깊이**: 메서드 바디 자체를 1단계로 보고, 그 안에 중첩되는 블록(`if`/`else if` 사슬 전체는
  하나의 단계로 취급, `for`/`while`/`try-catch` 포함)마다 1씩 증가시켜 계산한다.
  이 기준은 `ApprovalService.java` 클래스 상단 주석의 "중첩 if 6단계"와 동일한 산정 방식이다.
- **대상**: 실제 로직이 있는 클래스만 포함(`Service`, `Controller`, `Config`, `Seeder`).
  Entity(getter/setter만 존재)와 `JpaRepository` 인터페이스는 메서드 바디가 없어 제외.

## 2. 50줄 초과 메서드 상세 — `ApprovalService.processApproval()`

- **위치**: `ApprovalService.java` 75~160행 (86줄)
- **최대 중첩 깊이**: 6단계 — `승인(action=2)`/`반려(action=3)` 분기에서 발생

```
processApproval() {                                          # 1 (메서드 바디)
  if (d == null) return;                                     # 2
  if (u == null) return;                                     # 2
  if (proc == 1) { ... }                                      # 2  ── 상신
    if (s == 0) {                                              # 3
      if (type==1 && amount>=1000000) { ... }                   # 4
      if (approver != null) { ... }                             # 4
  } else if (proc == 2) { ... }                                # 2  ── 승인 (최대 깊이 지점)
    if (s == 1) {                                               # 3
      if (approverId != null && approverId.equals(userId)) {     # 4
        if (u.getRole() >= 2) {                                   # 5
          if (drafter != null) { ... }                             # 6  ← 최대 깊이
  } else if (proc == 3) { ... }                                # 2  ── 반려 (승인과 동일 구조, 동일 깊이)
    if (s == 1) {
      if (approverId != null && approverId.equals(userId)) {
        if (u.getRole() >= 2) {
          if (drafter != null) { ... }                             # 6  ← 최대 깊이
  } else if (proc == 9) { ... }                                # 2  ── 취소
    if (s == 0 || s == 1) {                                      # 3
      if (drafterId != null && drafterId.equals(userId)) { ... }   # 4
}
```

| 분기(action) | 줄 범위 | 최대 중첩 깊이 |
|---|---|---:|
| 상신(`proc==1`) | 90-109행 | 4 |
| 승인(`proc==2`) | 110-128행 | **6** |
| 반려(`proc==3`) | 129-148행 | **6** |
| 취소(`proc==9`) | 149-159행 | 4 |

승인/반려 두 분기가 상태 가드 → 결재자 본인 확인 → 권한 확인 → 메일 발송 대상 null 체크까지
4중으로 겹쳐 있어 메서드 전체 최대 깊이(6)를 만든다. 두 분기는 조건식이 사실상 동일해
([4-5. 중복 코드 탐지](./4-5.%20중복%20코드%20탐지.md) 4절 참고) 깊은 중첩 자체도 중복되는 셈이다.

기존 클래스 상단 주석은 "100줄 이상, 중첩 if 6단계"로 서술하지만, 실측 줄 수는 86줄이다
(주석은 스멜을 강조하기 위한 서술적 표현이며, 중첩 깊이 6단계는 실측과 정확히 일치한다).

## 3. 50줄은 안 넘지만 주목할 지점 — 깊은 중첩 대비 짧은 길이

| 파일 | 메서드 | 줄 수 | 중첩 깊이 | 비고 |
|---|---|---:|---:|---|
| `NoticeService.java` | `publish()` | 25 | **5** | `역할 판정 → 상태 가드 → 카테고리 판정 → 전 직원 반복문`이 25줄 안에 5단계로 압축되어 있음 |
| `ScheduleService.java` | `create()` | 39 | 4 | 검증 3연속 + 겹침 판정 반복문(`flag1`)이 중첩을 만듦 |
| `ScheduleService.java` | `confirm()` | 14 | 3 | 짧지만 소유자 확인 + 상태 가드가 중첩 |

`NoticeService.publish()` 는 줄 수(25줄)만 보면 짧아 보이지만, **줄당 중첩 밀도**가 코드베이스에서
가장 높다(25줄에 5단계) — 단순히 "50줄 초과"만 기준으로 보면 놓치기 쉬운 Long Method 계열 스멜이다.

## 4. 요약

- 50줄 초과 메서드: **`ApprovalService.processApproval()`** 1건 (86줄, 중첩 깊이 6) — Extract Method
  1순위 후보. action(1/2/3/9)별로 `submit()`/`approve()`/`reject()`/`cancel()` 4개 메서드로 분리하면
  각 메서드는 자연히 10~20줄, 중첩 깊이 3~4 수준으로 줄어든다.
- 중첩 깊이 4 이상 메서드: `processApproval()`(6), `NoticeService.publish()`(5), `ScheduleService.create()`(4)
  — 모두 "상태/권한 가드 → 실행 → 부가효과(메일·로그)"가 한 메서드 안에서 겹겹이 중첩된 동일한 구조.
- CLAUDE.md의 리팩토링 불변 규칙에 따라, 메서드를 쪼개더라도 `processApproval(id, userId, action, reason)`
  의 **public 시그니처는 유지**하고 내부에서 `submit`/`approve`/`reject`/`cancel`로 위임하는 방식이어야 한다.
