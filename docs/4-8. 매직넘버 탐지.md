# 4-8. 매직넘버 탐지

코드베이스 전체에서 정수 리터럴로 흩어져 있는 매직넘버를 필드/변수 단위로 모아, 각 값의 의미를
코드 주석과 실제 분기 로직을 근거로 추정한다. [4-5. 중복 코드 탐지](./4-5.%20중복%20코드%20탐지.md) 의
`role >= 2` 판정 중복, [4-7. 긴 메서드 탐지](./4-7.%20긴%20메서드%20탐지.md) 의 `processApproval()`
분기 구조와 함께 보면, 이 매직넘버들이 왜 Long Method·중복의 근본 원인인지 드러난다.

CLAUDE.md 불변 규칙에 따라 **DB 저장값(정수)은 리팩토링 후에도 그대로 유지**해야 하므로, 아래 표는
"enum으로 감쌀 코드 값이 무엇인가"를 확정하는 근거 자료로 사용한다.

## 1. 상태/유형/분류 코드값 (도메인 enum 후보)

### 1.1 `Approval.status`

| 값 | 추정 의미 | 근거 |
|---:|---|---|
| 0 | 임시저장(DRAFT) | `Approval.java` 26행 주석, `ApprovalService.create()` 53행 `setStatus(0)` |
| 1 | 상신(SUBMITTED) | `processApproval()` 97행 `setStatus(1)` — action=1(상신) 처리 결과 |
| 2 | 승인(APPROVED) | `processApproval()` 115행 `setStatus(2)` — action=2(승인) 처리 결과 |
| 3 | 반려(REJECTED) | `processApproval()` 134행 `setStatus(3)` — action=3(반려) 처리 결과 |
| 9 | 취소(CANCELED) | `processApproval()` 153행 `setStatus(9)` — action=9(취소) 처리 결과 |

`statusLabel()`(169-179행)의 라벨 매핑(임시저장/상신/승인/반려/취소)이 위 추정과 정확히 일치 —
가장 신뢰도 높은 근거.

### 1.2 `Approval.type`

| 값 | 추정 의미 | 근거 |
|---:|---|---|
| 1 | 지출 | `Approval.java` 25행 주석, `processApproval()` 94행 `d.getType() == 1` 분기(지출 결재만 금액 기준 우선순위 상향) |
| 2 | 휴가 | 주석만 존재, 서비스 로직에서 분기되는 곳 없음 |
| 3 | 구매 | 주석만 존재, 서비스 로직에서 분기되는 곳 없음 |
| 4 | 기타 | 주석만 존재, 서비스 로직에서 분기되는 곳 없음 |

**주의**: `type==1`(지출) 외 2/3/4는 코드 어디에서도 실제로 검사되지 않는다 — 주석에 적힌 의미가
정확한지 현재 로직만으로는 검증 불가(휴가/구매/기타를 구분해서 처리하는 분기가 아직 없음).

### 1.3 `Approval.priority`

|---:|---|---|
| 1 | 낮음 | `Approval.java` 27행 주석 |
| 2 | 보통 | 동상 |
| 3 | 높음 | `create()` 52행 `urgent ? 3 : priority`, `processApproval()` 95행 `setPriority(3)`(고액 지출 자동 상향) — 둘 다 "3=중요/긴급"으로 쓰임, 주석과 일치 |

| 3 | 인사 | 주석만 존재, 서비스 로직에서 분기되는 곳 없음 |

### 1.6 `Schedule.status`

| 값 | 추정 의미 | 근거 |
|---:|---|---|
| 0 | 예정 | `Schedule.java` 18행 주석, `ScheduleService.create()` 58행 `setStatus(0)` |

### 1.7 `User.role`

| 값 | 추정 의미 | 근거 |
|---:|---|---|
| 1 | 사원 | `User.java` 19행 주석, `DataSeeder` 24행(김사원)·27행(최사원) |
| 2 | 팀장 | `DataSeeder` 25행(박팀장). `role >= 2` 로만 쓰여 "2"라는 정확한 값 자체는 결재 승인/반려·공지 게시 권한의 **경계값**으로만 의미가 있다 |
| 3 | 임원 | `DataSeeder` 26행(이임원). 로직상 2(팀장)와 구분해서 별도 처리하는 분기는 없음 — "팀장 이상"으로만 취급됨 |

### 1.8 결재 처리 액션 코드 (`processApproval` 의 `action` 파라미터, 필드 아님)

| 값 | 추정 의미 | 근거 |
|---:|---|---|
| 1 | 상신 | `ApprovalController.java` 36행 주석, `processApproval()` 90행 `proc == 1` |
| 2 | 승인 | 110행 `proc == 2` |
| 3 | 반려 | 129행 `proc == 3` |
| 9 | 취소 | 149행 `proc == 9` (4~8은 정의되지 않은 채 비어 있음) |

**발견**: action 코드 값이 우연이 아니라 **처리 결과로 설정되는 status 값과 1:1로 동일**하다
(action=1→status=1, action=2→status=2, action=3→status=3, action=9→status=9). 이 대응 관계 덕분에
"action 코드가 왜 9인가"를 status 쪽 의미(9=취소)에서 역으로 추정할 수 있었다.

## 2. 도메인 간 공통 관례 — "0/1/9" 패턴

세 엔티티의 `status` 필드를 나란히 놓으면 코드 값의 자리마다 공통된 관례가 드러난다.

| 코드 | Approval | Notice | Schedule |
|---:|---|---|---|
| 0 | 임시저장 | 임시 | 예정 |
| 1 | 상신 | 게시 | 확정 |
| 2 | 승인 | (없음) | (없음) |
| 3 | 반려 | (없음) | (없음) |
| 9 | 취소 | 내림 | 취소 |

→ **0 = 초안/시작 상태, 1 = 다음 단계로 진행, 9 = 최종 종료(취소/내림)** 라는 암묵적 규칙이 세 도메인에
공통으로 적용되어 있다. `Approval` 만 2(승인)/3(반려) 중간 상태를 추가로 가진다. 이 규칙은 코드
어디에도 명시적으로 문서화되어 있지 않고, 세 엔티티를 비교해야만 드러난다 — enum화 시 도메인마다
별도 enum을 두더라도 "9=종료" 관례는 유지하는 편이 기존 데이터/직관과 맞는다.

## 3. 금액 임계값 매직넘버

| 리터럴 | 추정 의미 | 위치 |
|---:|---|---|
| `1_000_000` (100만원) | 고액 지출 결재 자동 우선순위 상향 기준 | `ApprovalService.processApproval()` 94행 — `type==1(지출) && amount>=1000000` → `priority=3` |
| `10_000_000` (1000만원) | 금액 등급 "S" 기준 | `ApprovalService.amountGrade()` 184행 |
| `1_000_000` (100만원) | 금액 등급 "A" 기준 | `amountGrade()` 185행 |
| `100_000` (10만원) | 금액 등급 "B" 기준 | `amountGrade()` 186행 |

**주의**: `1,000,000`(100만원) 이라는 동일한 리터럴이 `processApproval()`의 우선순위 상향 기준과
`amountGrade()`의 A등급 기준 두 곳에 각각 독립적으로 등장한다. 두 값이 "100만원은 중요한 금액"이라는
같은 개념을 의도적으로 공유하는 것인지, 우연히 같은 숫자를 썼을 뿐인지는 코드만으로는 판단할 수 없다.
enum·상수화 시 **하나로 묶을지 별도 상수로 분리할지 결정이 필요한 지점**이다.

## 4. 기타 매직넘버

| 리터럴 | 추정 의미 | 위치 |
|---:|---|---|
| `2` | MariaDB 연결 가능 여부 확인 타임아웃(초) | `DataSourceConfig.isMariaDbReachable()` 51행 `DriverManager.setLoginTimeout(2)` — 이미 `MARIADB_URL/USER/PASSWORD` 는 상수화되어 있으나 타임아웃 값만 리터럴로 남아 있음 |

## 5. 요약 — enum화 대상 정리

| 필드/파라미터 | 값 범위 | 코드 내 실제 분기 여부 |
|---|---|---|
| `Approval.status` | 0/1/2/3/9 | 5개 값 모두 실사용(가장 명확) |
| `Approval.type` | 1/2/3/4 | `1`만 실사용, 2/3/4는 주석에만 존재 |
| `Approval.priority` | 1/2/3 | `3`만 명시적 설정 로직 존재(자동 상향), 1/2는 요청값 그대로 통과 |
| `Notice.status` | 0/1/9 | 0/1만 실사용, 9는 주석에만 존재(전이 로직 없음) |
| `Notice.category` | 1/2/3 | `2`만 실사용 |
| `Schedule.status` | 0/1/9 | 0/1/9 모두 사용(9는 겹침 검사 제외 조건으로) |
| `User.role` | 1/2/3 | 경계값 `>=2` 만 사용, 1/2/3 개별 값 분기는 없음 |
| `action`(파라미터) | 1/2/3/9 | 4개 값 모두 실사용, status와 1:1 대응 |

CLAUDE.md 방향에 따라 각 필드는 별도 enum + `AttributeConverter` 로 DB 정수값을 유지한 채 감싸는
것이 원칙이나, 실제 분기에서 쓰이지 않는 값(`Approval.type`의 2/3/4, `Notice.category`의 1/3,
`Notice.status`의 9)은 enum 상수 자체는 정의하되 **아직 검증된 동작이 없다**는 점을 리팩토링 시
주석으로 남겨야 한다(보존 대상과 미검증 대상을 혼동하지 않기 위함).
