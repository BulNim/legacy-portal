# 리팩토링 효과 분석

`refactor/start` 태그(특성화 테스트 완료 시점) 이후 `ApprovalService` 계열에 적용한 리팩토링
(enum + AttributeConverter, `processApproval` guard clause·메서드 추출·switch 전환)의 효과를
정량·정성으로 분석한다.

## 1. 구조 지표 (before → after)

| 지표 | 리팩토링 전 (`refactor/start`) | 현재 | 효과 |
|---|---|---|---|
| `processApproval` 길이 | **86줄** 단일 메서드 | ~15줄 dispatcher + 4개 메서드(각 24~25줄) | 한 덩어리 → 책임 분리 |
| `processApproval` 최대 중첩 깊이 | **6단계** (중첩 if 지옥) | **2단계** (guard clause) | 가독성 급상승 |
| 분기 방식 | `if/else if` 4연쇄 + 중첩 | `switch` + guard clause | 의도 명확 |
| `statusLabel` | 11줄 (tmp + 5분기 if) | **1줄** (enum 위임) | 중복 라벨표 제거 |
| processApproval 계열 매직넘버 | **17개** bare 정수 리터럴 | **0개** (전부 enum) | 의미 코드에 내재화 |

## 2. 매직넘버 제거 상세

- `status` 0/1/2/3/9, `action` 1/2/3/9, `type` 1, `priority` 3, `role` 2 →
  전부 `ApprovalStatus`/`ApprovalAction`/`ApprovalType`/`Priority`/`Role` enum 상수로.
- 필드는 enum + `AttributeConverter` → **DB엔 정수 그대로 저장**(관찰 동작 불변).
- 의도적으로 남긴 것: 금액 임계값 `1000000`, `amountGrade()`의 등급 기준(1000만/100만/10만) —
  범주형이 아닌 수치라 enum 대상이 아님.

## 3. 안전성 — 가장 중요한 효과

**전 과정에서 특성화 테스트 6개가 byte 단위로 무변경, 계속 green.**

| 리팩토링 단계 | 테스트 |
|---|:---:|
| enum + AttributeConverter 전환 | 6/6 green (무수정) |
| guard clause + 메서드 추출 | 6/6 green (무수정) |
| switch 전환 | 6/6 green (무수정) |

→ 겉보기 계약(시그니처·DB·JSON·동작)이 100% 보존됐음이 자동 검증됨.
"리팩토링이 맞다(동작 변경이 아니다)"는 것을 테스트가 증명한다.

## 4. 비용 (트레이드오프)

- **파일 9개 신규**: enum 5개 + converter 4개. 대신 각 파일은 단일 책임·소형이고, 흩어져 있던
  매직넘버·라벨표가 한곳으로 모였다.
- 코드 총량: 수정 파일 기준 +175/−128줄(enum/converter 신규분 별도). 줄 수는 늘었지만 의미 밀도·안전성이 오름.

## 5. 종합 판단 — "안전한 출발"이지 "충분한 효과"는 아직 아니다

| 질문 | 답 |
|---|---|
| 리팩토링이 **안전**했나? | ✅ 예 (특성화 테스트가 증명) |
| **읽기** 좋아졌나? | ✅ 예 |
| **변경하기** 좋아졌나(유지보수성)? | △ 부분적 — 중복·강결합이 남아 아직 아님 |
| 실무에서 "이만하면 됐다"인가? | ❌ 아직 — God Class가 핵심인데 안 건드림 |

핵심 문제는 매직넘버·중첩이 아니라 **God Class**다. `ApprovalService`는 아직도 영속화 + 메일 발송 +
감사 로그 + 권한 판정 + 화면 라벨 생성을 혼자 다 한다. 메서드를 4개로 쪼갰지만, 쪼갠 4개가 여전히
**같은 중복(메일 본문 조립·권한 판정 조건식)을 각자 안고** 있어 결합·중복이라는 근본 원인은 그대로다.
지금 "메일 문구 바꿔라"가 오면 여전히 4곳을 고쳐야 한다 → "변경 용이성" 효과는 아직 체감되지 않는다.
